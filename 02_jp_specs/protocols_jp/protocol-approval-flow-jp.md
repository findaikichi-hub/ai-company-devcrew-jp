# protocol-approval-flow-jp

## 目的
日本企業における承認状態遷移、差戻し条件、再申請条件、停止条件、例外承認接続を、AI社員運用で破綻なく扱うための protocol 骨格を定義する。

## 位置づけ
本ファイルは、承認フロー全体の状態遷移を扱う protocol 別仕様の第二稿とする。  
本ファイルが扱うのは、承認状態・遷移条件・権限確認・停止条件である。  
起案内容そのもの、必要添付そのものの妥当性確認は `protocol-ringi-jp.md` 側で扱う。

## 扱う対象
- 承認開始条件
- 承認状態の定義
- 状態遷移条件
- 差戻し条件
- 再申請条件
- 停止条件
- 例外承認条件
- 状態起因 / 権限起因の HITL

## 入力
本ファイルで扱う入力カテゴリは以下。
- 承認対象情報
- 現在状態
- 差戻し理由
- 再申請有無
- 権限確認結果
- 例外承認要否
- 前段 protocol からの接続結果

## 出力
本ファイルで扱う出力カテゴリは以下。
- 次状態
- 差戻し判定
- 再申請待ち判定
- 停止 / 保留判定
- 例外承認分岐判定
- HITL 接続要求

## 前提条件
本 protocol を適用する前提は以下。
- 起案内容が承認接続可能な状態にある
- 承認対象が識別できる
- 状態遷移判断に必要な前提情報が揃っている
- 承認可否または差戻し可否の対象が明確である
- 権限確認の必要性が認識されている

## 基本原則
- 承認フローは state transition として扱う。
- 未承認状態のまま完了扱いしない。
- 差戻し理由は追跡可能に残す。
- 例外承認は通常承認と分けて扱う。
- 承認権限が不明な場合は自律的に前進しない。
- 部署名、役職名、具体的承認経路、期限固定値は本ファイルで確定しない。

## 承認開始条件
承認フローは、少なくとも以下を満たす場合に開始する。
- 起案内容が承認接続可能な状態にある
- 承認対象が識別できる
- 状態遷移を開始する前提情報が揃っている
- 承認判断または差戻し判断の対象が明確である
- 承認権限の確認が必要であることが明らかである

## 承認状態の定義
本第二稿で扱う基本状態は以下。
- 未申請
- 申請済み
- 一次確認待ち
- 承認待ち
- 差戻し
- 再申請待ち
- 例外承認待ち
- 承認済み
- 却下
- 保留

## 状態遷移条件
基本的な遷移条件の骨格は以下。
- 未申請 → 申請済み  
  起案内容が承認接続可能となった場合
- 申請済み → 一次確認待ち  
  初期確認が必要な場合
- 一次確認待ち → 承認待ち  
  初期確認で前進可能と判断された場合
- 承認待ち → 承認済み  
  承認可否の判断が完了した場合
- 承認待ち → 差戻し  
  状態上または条件上の不足が確認された場合
- 差戻し → 再申請待ち  
  差戻し理由が通知され、再申請待ちに入る場合
- 再申請待ち → 申請済み  
  再提出が行われた場合
- 承認待ち → 例外承認待ち  
  通常経路で処理できず、例外判断が必要な場合
- 任意状態 → 保留  
  前進判断に必要な条件が欠けている場合
- 承認待ち / 例外承認待ち → 却下  
  明確に前進不可と判断された場合

## 差戻し条件
差戻しは、主として状態遷移上の前提不足または条件未充足に起因して行う。  
差戻し条件の骨格は以下。
- 情報不足
- 添付不足
- 条件未充足
- 権限不一致
- 申請区分誤り

差戻し理由は、再申請可能な粒度で残す。

## 再申請条件
再申請は、差戻し理由に対する補完または修正が行われた場合に進める。  
再申請条件の骨格は以下。
- 差戻し理由が明示されている
- 不足条件が解消されている
- 承認対象が再度識別可能である
- 再申請後に再審査可能な状態である

## 停止条件
以下の場合は、承認フローを進めず停止または保留する。
- 承認権限が曖昧
- 承認対象が不明
- 通常経路で処理可否を判断できない
- 必要条件未充足のまま前進しようとしている
- 人間判断なしでは誤処理リスクが高い

## 例外承認条件
例外承認へ分岐する条件の骨格は以下。
- 通常経路外の承認が必要
- 緊急対応時の承認が必要
- 一時的運用例外が必要
- 権限委譲を伴う承認が必要
- 通常条件では処理不能だが、停止のみでは不足する場合

## 非対象
本ファイルの非対象は以下。
- 起案内容そのものの妥当性確認
- 必要添付そのものの妥当性確認
- 部署名、役職名、具体経路、金額閾値
- 承認権限規程そのもの
- 個人情報 / 外部共有の統制詳細
- 監査証跡 artifact の完全フォーマット
- Orchestrator の全体進行管理

## HITL 条件
人間介入が必要となる条件の骨格は以下。
- 承認可否の最終判断が必要
- 例外承認の許可が必要
- 差戻し判断の確定が必要
- 権限曖昧時に前進停止判断が必要
- 状態遷移だけでは処理可否を決められない

## HITL トリガー
HITL 条件が実際に発火する契機の骨格は以下。
- 権限確認不能
- 状態遷移条件の競合
- 通常経路外の承認要求
- 却下 / 保留 / 例外承認の選択が必要
- 前進判断に必要な根拠が不足している

## HITL 制御
HITL 発火後の接続点の骨格は以下。
- 人間確認待ちへ遷移する
- 承認待ちから保留または例外承認待ちへ切り替える
- 差戻しまたは却下へ分岐する
- 人間判断結果を前提に状態遷移を再開する
